1. Overview

This report documents the refactoring of the GMM estimation pipeline in preamble.R. The goal was to eliminate code duplication by having the GMM objective functions use the already-refactored core model functions.

What is this code doing?

This estimation pipeline estimates a worker-task assignment model for the salon industry. The model determines how different types of workers (e.g., by skill level) are allocated to different tasks (e.g., haircuts, coloring, styling) within firms.

The key economic insight is that firms face a trade-off between specialization (assigning workers to tasks they're best at) and flexibility (having workers who can do multiple tasks). The parameter γ (gamma) captures the organizational cost of specialization.

3.2 build_cost_matrix()

Constructs the cost matrix that determines worker-task assignment preferences:

build_cost_matrix <- function(theta, beta, config = CONFIG) {
  # theta: wage premium parameters (what we're estimating)
  # beta: skill parameters from first-stage IV regression

  for (cnty in config$counties) {
    # Step 1: Build wage matrix
    # Worker type 1 is the reference (wage = 0)
    # Other types have wage premiums from theta
    w_mat <- matrix(c(0, w_params), ncol = n_t, nrow = n_w, byrow = FALSE)

    # Step 2: Extract skill coefficients from beta
    # These come from the demand-side IV regression
    skills <- matrix(skill_params, ncol = n_t, nrow = n_w, byrow = FALSE)

    # Step 3: Compute cost matrix
    # cost[i,j] = wage_premium[i] + skill[i,j] / rho
    # rho is price sensitivity (typically negative)
    tild_theta[[cnty]] <- w_mat + (1/rho) * skills

    # Step 4: Normalize so min cost per task = 0
    # This is a normalization that doesn't affect results
    tild_theta[[cnty]] <- sweep(tild_theta[[cnty]], 2, col_mins, "-")
  }
  return(tild_theta)
}
Economic Interpretation: The cost matrix captures how "expensive" it is to assign worker type i to task type j. Lower cost means the worker-task pair is more productive (either because of lower wages or higher skill match).

3.3 solve_equilibrium()

Finds the equilibrium worker allocation given a regularization parameter γ:

solve_equilibrium <- function(cost_matrix, alpha, gamma, innertol, config) {
  # cost_matrix: the tild_theta matrix for one county
  # alpha: task mix (what share of work is each task type)
  # gamma: regularization parameter (org cost)

  # Step 1: Compute preference matrix A
  # A[i,j] = exp(-cost[i,j] / gamma)
  # Higher gamma = more mixing allowed
  # Lower gamma = more specialization

  # Use log-space for numerical stability:
  log_A <- -cost_matrix / gamma
  log_A_shifted <- log_A - max(log_A)  # Shift prevents overflow
  A <- exp(log_A_shifted)

  # Step 2: Fixed-point iteration to find E (worker shares)
  # The equilibrium condition is:
  # E[i] = E[i] * sum_j(A[i,j] * alpha[j] / sum_k(A[k,j] * E[k]))

  fxpt <- function(p) {
    col_sums <- A_t %*% p           # sum_k(A[k,j] * p[k]) for each j
    task_scales <- alpha / col_sums  # alpha[j] / sum_k(...)
    update <- A %*% task_scales      # sum_j(A[i,j] * task_scales[j])
    return(p * update)
  }

  # Use SQUAREM for accelerated convergence
  result <- squarem(E, fixptfn = fxpt, ...)
  E <- result$par
  E <- E / sum(E)  # Normalize to sum to 1

  # Step 3: Compute assignment matrix B
  # B[i,j] = share of task j done by worker type i
  B <- A * E                          # Multiply rows by E
  B <- t(t(B) * alpha / col_sums)    # Scale columns

  # Step 4: Compute entropy (specialization index)
  # entropy = sum_ij B[i,j] * log(B[i,j] / (E[i] * alpha[j]))
  B_rel <- B / E / alpha  # Relative assignment
  entropy <- sum(B * spec_log(B_rel))

  return(list(E = E, B = B, entropy = entropy, ...))
}
Equilibrium condition: Bij = Aij · αj / Σk(Akj · Ek) · Ei

3.4 find_gamma_for_sindex()

Uses bisection to find the γ that produces a target entropy level:

find_gamma_for_sindex <- function(cost_matrix, alpha, s_index, innertol, outertol, config) {
  # s_index: target entropy (observed specialization level)

  # Step 1: Check if corner solution is needed
  # Corner = maximum specialization (each task done by one worker type)
  corner <- compute_corner_solution(cost_matrix, alpha, config)

  if (s_index > corner$entropy_bound) {
    # Requested specialization exceeds what's achievable
    # Return corner solution with gamma = 0
    return(list(gamma = 0, E = corner$E, B = corner$B, is_corner = TRUE))
  }

  # Step 2: Define objective for bisection
  # We want: entropy(gamma) = s_index
  # So we find root of: f(gamma) = s_index - entropy(gamma)
  objective <- function(gamma) {
    eq <- solve_equilibrium(cost_matrix, alpha, gamma, innertol, config)
    return(s_index - eq$entropy)
  }

  # Step 3: Run bisection search
  result <- bisection(objective, a = 1, b = 10000, ...)

  # Step 4: Compute final equilibrium at found gamma
  gamma_final <- result$root
  eq_final <- solve_equilibrium(cost_matrix, alpha, gamma_final, innertol, config)

  return(list(gamma = gamma_final, E = eq_final$E, B = eq_final$B, ...))
}
Key Insight: The relationship between γ and entropy is monotonic:
  • γ → 0: Maximum specialization, entropy → entropy_bound
  • γ → ∞: Maximum mixing (uniform), entropy → 0
This monotonicity allows us to use bisection to find the γ matching any target entropy.

4.1 get_worker_allocation() NEW

A new helper function that wraps find_gamma_for_sindex:

#' Compute worker allocation for a single observation
#' Returns E[2:5] (excludes E_1 for identification)
get_worker_allocation <- function(cost_matrix, alpha, s_index,
                                    innertol, outertol, config = CONFIG) {
  # Call the core function to get full equilibrium
  result <- find_gamma_for_sindex(cost_matrix, alpha, s_index,
                                   innertol, outertol, config)

  # Return only E[2:5] (worker types 2-5)
  # E[1] is excluded to avoid collinearity (shares sum to 1)
  return(result$E[2:config$n_worker_types])
}

objective_gmm <- function(theta, x) {
  # theta: wage parameters we're trying to estimate
  # x: data frame with observations

  data <- x
  pre_parms <- theta
  names(pre_parms) <- names(beta_2_subset)  # Apply parameter names

  # Build cost matrices for all counties at once
  # This is done ONCE per function call, not per observation
  tild_theta <- build_cost_matrix(pre_parms, beta, CONFIG)

  # Helper: compute worker shares for one observation
  get_demands <- function(a1, a2, a3, a4, a5, county, s_index) {
    alpha <- c(a1, a2, a3, a4, a5)  # Task mix for this firm
    # Call the core function - no duplicated solver logic!
    get_worker_allocation(tild_theta[[county]], alpha, s_index,
                           innertol, outertol, CONFIG)
  }

  # Wrapper for parallel execution
  get_demands_outer <- function(idx) {
    get_demands(
      data$task_mix_1[idx], data$task_mix_2[idx], data$task_mix_3[idx],
      data$task_mix_4[idx], data$task_mix_5[idx],
      data$county[idx], data$s_index[idx]
    )
  }

  # Run in parallel (Windows) or with mclapply (Unix)
  if (pl_on == TRUE) {
    if (get_os() == "windows") {
      temp_res <- data.table(do.call(rbind,
        parLapply(clust, 1:nrow(data), get_demands_outer)))
    } else {
      temp_res <- data.table(do.call(rbind,
        mclapply(1:nrow(data), get_demands_outer, mc.cores = core_count)))
    }
  } else {
    temp_res <- data.table(do.call(rbind,
      lapply(1:nrow(data), get_demands_outer)))
  }

  # Build moment conditions: (model E - observed E)
  # This is the "error" that GMM tries to minimize
  E_match <- data.frame(
    (as.matrix(temp_res) - as.matrix(data[, c("E_raw_2", "E_raw_3",
                                                    "E_raw_4", "E_raw_5")])),
    factor(data[, "county"])
  )
  colnames(E_match) <- c("E_2", "E_3", "E_4", "E_5", "county")

  # Create moment matrix with county interactions
  E_mat <- model.matrix(~county:E_2 + county:E_3 + county:E_3 +
                          county:E_4 + county:E_5 - 1, data = E_match)

  return(E_mat)
}

5. Logic Equivalence Analysis

A careful comparison was done to ensure the refactored code produces identical results.

5.1 Cost Matrix Construction

✔ Equivalent

Both versions compute: w_mat + (1/rho) * skills

5.2 A Matrix Computation

Aspect Original Refactored
Formula exp(-cost/gamma) exp(-cost/gamma - max)
Purpose Direct computation Log-space shift for stability
Mathematical result Identical (shift cancels in ratios)
Why the shift is safe: A is only used in ratios like A[i,j] / sum(A[k,j] * E[k]). If we multiply all A values by a constant c, it cancels: (c*A[i,j]) / sum(c*A[k,j] * E[k]) = A[i,j] / sum(A[k,j] * E[k])

5.3 Fixed-Point Function

✔ Equivalent

Both compute: E_new[i] = E[i] * Σj(A[i,j] * α[j] / Σk(A[k,j] * E[k]))

5.4 Key Differences (Improvements)

Feature Original Refactored
E normalization None after SQUAREM E <- E / sum(E)
Division by zero Can produce NaN pmax(x, 1e-16) protection
Overflow prevention Post-hoc clamping Log-space computation

6. Numerical Stability Improvements

The refactored code includes several numerical safeguards:

6.1 Log-Space A Matrix Computation

# Problem: exp(-cost/gamma) can overflow when cost/gamma is large negative

# Original approach:
A <- exp(-cost_matrix / gamma)
A[A >= Inf] <- 1e16  # Fix after the fact

# Refactored approach:
log_A <- -cost_matrix / gamma
log_A_shifted <- log_A - max(log_A)  # Shift so max is 0
A <- exp(log_A_shifted)              # Now exp(0) = 1 is the max

6.2 Protected Division

# Problem: Division by zero when col_sums contains zeros

# Original: No protection
task_scales <- alpha / colSums(A * p)  # Can be Inf or NaN

# Refactored: Floor protection
col_sums <- pmax(col_sums, config$numeric_floor)  # At least 1e-16
task_scales <- alpha / col_sums  # Safe division

6.3 E Normalization

# Problem: SQUAREM may not perfectly preserve sum(E) = 1

# Original: Trust SQUAREM output directly
E <- squarem(...)$par

# Refactored: Ensure normalization
E <- squarem(...)$par
E <- pmax(E, config$numeric_floor)  # Ensure positive
E <- E / sum(E)                     # Normalize to sum to 1
